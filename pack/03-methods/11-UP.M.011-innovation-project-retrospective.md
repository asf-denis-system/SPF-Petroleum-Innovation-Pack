# [UP.M.011] Ретроспективный анализ инновационного проекта

## Определение

**Метод** - структурированный разбор завершенного НИР/НИОКР/ОПР, который определяет, где именно порвалась связка `проблема -> технология -> испытание -> внедрение -> эффект`, и превращает завершение проекта в повторно используемое знание.

Этот метод специально отделяет:
- неуспех самой технологии;
- неуспех постановки задачи;
- неуспех выбора объекта и дизайна испытаний;
- неуспех внедрения, принятия или измерения эффекта.

## Цель

1. **Не сводить разбор к вопросу "кто виноват"**, а локализовать разрыв по стадиям и артефактам.
2. **Сохранить знание**, даже если гипотеза не подтвердилась.
3. **Вернуть уроки обратно в конвейер проектов и технологий**, чтобы следующий цикл был сильнее.

## Когда применять

- После завершения НИР, НИОКР, ОПИ или ОПР.
- После остановки пилота по `NO-GO` или `PIVOT`.
- После частичного успеха, когда нужно определить границы применимости и решение по следующему циклу.

## Процесс

### Этап 1: Зафиксировать итоговый статус проекта

Использовать не бинарную логику, а один из 5 статусов:

| Итоговый статус | Как трактовать |
|---|---|
| **Успех подтвержден** | Эффект подтвержден и есть основание для внедрения или тиража |
| **Частичный успех** | Эффект есть, но только для части условий, объектов или режимов |
| **Гипотеза не подтвердилась, но знание получено** | Метод не дал эффекта, зато сузил область применимости и сократил будущие потери |
| **Результат не доказан** | Не удалось честно доказать эффект из-за данных, дизайна испытаний или методики расчета |
| **Организационный срыв** | Проект сорвался по управлению, ресурсам, handover или приоритетам, а не по технологии как таковой |

### Этап 2: Найти ключевой разрыв в цепочке

Проверить, где именно сломался проект:

| Этап цепочки | Типовой вопрос |
|---|---|
| **Приоритизация проблемы** | Работали ли под реальную приоритетную проблему и понятного владельца? |
| **Выбор технологии/подхода** | Соответствовала ли технология состоянию системы? |
| **Инициация проекта** | Были ли заданы KPI, baseline, объекты, роли и бюджет? |
| **Разработка технологии** | Была ли технология доведена до устойчивого состояния? |
| **Подготовка испытаний** | Был ли корректный дизайн испытаний и план измерений? |
| **Тестирование / ОПИ / ОПР** | Были ли объект и условия пилота репрезентативны? |
| **Анализ результатов** | Можно ли честно доказать или опровергнуть эффект? |
| **Принятие решения** | Было ли принято ясное решение по дальнейшему циклу? |
| **Упаковка / внедрение / тираж** | Сработал ли handover в эксплуатацию и масштабирование? |

### Этап 3: Классифицировать причины, а не искать одну "главную"

Использовать причинную сетку, а не одну линейную причину:

| Уровень причины | Что проверяем |
|---|---|
| **Проблема / целеполагание** | Приоритетность проблемы, наличие владельца, критерии успеха |
| **Гипотеза / концепция** | Физическое или технико-методическое обоснование идеи |
| **Объект / применимость** | Корректность выбора объекта, условий, данных, окна наблюдения |
| **Разработка / исполнитель** | Готовность решения, качество подрядчика, полнота комплекта |
| **Испытания / данные / расчет** | Доказательность дизайна, качество данных, корректность атрибуции эффекта |
| **Организация / ресурсы** | Команда, приоритет, handover, контрольные точки, ритм управления |
| **Внедрение / принятие / масштаб** | Принятие эксплуатацией, тиражируемость, экономика, устойчивость |

### Этап 4: Применить "5 почему" только к каждому ключевому разрыву

`5 почему` здесь полезен не для всего проекта сразу, а для одного локализованного разрыва. Иначе метод насильно сводит составную проблему к одной причине и теряет системность.

### Этап 5: Отделить "технология не сработала" от "мы не доказали результат"

Ключевое правило: статус **"результат не доказан"** не равен статусу **"технология не работает"**. Если нет корректного baseline, плана измерения и окна наблюдения, то ретроспектива должна честно фиксировать недоказанность, а не ложный `NO-GO`.

### Этап 6: Выпустить решение по следующему циклу

Типовые решения:

| Решение | Когда подходит |
|---|---|
| **Закрыть** | Гипотеза и область применения исчерпаны, дальнейший цикл нецелесообразен |
| **Доработать и повторить** | Есть сильная идея, но нужны изменения технологии или организационного контура |
| **Повторить испытание** | Главный разрыв был в дизайне испытаний или данных |
| **Ограничить область применения** | Эффект есть только для определенного класса объектов |
| **Готовить к внедрению / тиражу** | Эффект подтвержден и границы применимости понятны |

## Инструменты

| Инструмент | Для чего |
|---|---|
| **Шаблон ретроспективы проекта** | Единая форма фиксации результата и причин |
| **Классификатор причин** | Сравнимость проектов и агрегирование повторяющихся разрывов |
| **Протокол эффекта / baseline** | Разделение реального эффекта и недоказанности |
| **Gate-лог** | Трассировка решений и допущений по ходу проекта |

## Пример

**Ситуация:** ОПР новой технологии обработки призабойной зоны дал эффект только на 3 из 8 скважин.

**Неправильный вывод:** "технология плохая".

**Правильный вывод по методу:**
- итоговый статус: **частичный успех**;
- ключевой разрыв: **объект / применимость**;
- вторичный разрыв: **испытания / данные / расчет**;
- знание: технология работает только для подмножества коллекторов и требует более жесткого фильтра выбора кандидатов;
- решение: **ограничить область применения** и обновить критерии отбора объектов.

## Когда метод сложен

- Когда проект смешивает исследование, пилот и внедрение без явных gate-точек.
- Когда нет baseline и невозможно честно отделить эффект от шума.
- Когда в проекте одновременно несколько частичных результатов и несколько конкурирующих причин.

## Связанные принципы

- **OA.IP.001** - Технология не равна эффекту.
- **OA.IP.002** - Лаборатория и поле неэквивалентны.
- **OA.IP.003** - Инновация увеличивает вариативность и требует аккуратной квалификации.
- **OA.OP.002** - Информация теряется при передаче между ролями и стадиями.

## Связанные failure modes

- **FM.IP.001** - Копирование технологии без проверки применимости.
- **FM.IP.002** - Перенос лабораторной правды в поле без промежуточной квалификации.
- **FM.IP.003** - Масштабирование без достаточной квалификации.
- **FM.ORG.002** - Потеря критической информации при handover.
