# [UP.M.013] Problem-first приоритезация инновационных проектов

## Определение

**Метод** - приоритизация инновационных проектов, в которой сначала оценивается **важность проблемного поля**, а затем - **сила и готовность конкретного решения**. Это защищает портфель от двух ошибок:

- в топ попадают "красивые", но не самые нужные технологии;
- важные проблемные поля теряют место, потому что по ним пока нет зрелого проекта.

Итоговая логика такая:

`итоговый приоритет = приоритет проблемы + качество решения + готовность к реализации`

точнее:

`final_priority = problem_score * 0.40 + project_score * 0.60`

## Цель

1. **Сначала отдать приоритет проблеме**, а не технологическому энтузиазму.
2. **Развести проекты прямого воздействия и обеспечивающие проекты**, чтобы они не сравнивались по одной и той же шкале.
3. **Разложить портфель по горизонтам и корзинам приоритета**, чтобы длинные технологии не проигрывали коротким автоматически.

## Когда применять

- На стадии `Приоритизация` конвейера проектов.
- При подготовке портфеля проектов для комитета / ГД.
- При сравнении кандидатов внутри одной производственной темы или проблемного поля.

## Входные данные

| Входные данные | Описание |
|---|---|
| **Паспорт проекта** | Единая карточка проекта: тип, проблема, объект, стадия, горизонт, ожидаемый эффект, доказанность |
| **Описание проблемного поля** | Масштаб потерь, охват активов, срочность, стратегическая значимость |
| **Класс проекта** | Прямое производственное воздействие или обеспечивающий проект |
| **Данные о реализуемости** | Полигон, подрядчик, ресурсы, готовность к запуску |
| **Горизонт эффекта** | `H1`, `H2`, `H3` |

## Процесс

### Этап 1: Нормализовать проекты в единый паспорт

Для каждого проекта должны быть зафиксированы:
- `id`;
- название;
- тип проекта;
- класс технологии;
- проблемное поле;
- тип объекта;
- стадия;
- горизонт эффекта;
- ожидаемый эффект;
- масштаб применения;
- наличие полигона;
- степень доказанности.

Без этого проекты не сравнимы.

### Этап 2: Оценить приоритет проблемного поля

Используется шкала `1..5` и веса:

| Критерий проблемы | Вес | Смысл |
|---|---|---|
| Масштаб проблемы | 30% | Какой объем потерь / недобора стоит за проблемой |
| Охват активов | 20% | На скольких объектах и активах проблема проявляется |
| Срочность | 20% | Насколько важно решать проблему в горизонте 1-3 лет |
| Стратегическая значимость | 20% | Насколько проблема связана с ключевыми целями компании |
| Потенциал эффекта | 10% | Насколько велик верхний потолок выигрыша |

Формула:

`problem_score = scale * 0.30 + coverage * 0.20 + urgency * 0.20 + strategic * 0.20 + potential * 0.10`

### Этап 3: Разделить проекты на 2 класса

#### Тип A: проекты прямого производственного воздействия

Примеры:
- химические, газовые, тепловые МУН;
- ВИР;
- интенсификация;
- управление заводнением, если оно дает прямой производственный эффект.

#### Тип B: обеспечивающие проекты

Примеры:
- исследования;
- диагностика;
- трассеры;
- модели;
- ПО;
- методы, повышающие качество выбора решения.

### Этап 4: Оценить проект по правильной модели

#### 4.1. Для прямого воздействия

| Критерий проекта | Вес |
|---|---|
| Ожидаемый производственный эффект | 30% |
| Масштабируемость | 20% |
| Срок до результата | 15% |
| Доказанность | 15% |
| Реализуемость | 10% |
| Соответствие типу объекта | 10% |

Формула:

`project_score_direct = effect * 0.30 + scale_out * 0.20 + speed * 0.15 + evidence * 0.15 + readiness * 0.10 + object_fit * 0.10`

#### 4.2. Для обеспечивающих проектов

| Критерий проекта | Вес |
|---|---|
| Влияние на качество решений | 30% |
| Снижение неопределенности | 25% |
| Охват объектов / решений | 20% |
| Скорость встраивания в практику | 10% |
| Доказанность метода | 10% |
| Реализуемость | 5% |

Формула:

`project_score_enabling = decision_quality * 0.30 + uncertainty_reduction * 0.25 + coverage * 0.20 + embed_speed * 0.10 + evidence * 0.10 + readiness * 0.05`

### Этап 5: Посчитать итоговый приоритет проекта

`final_priority = problem_score * 0.40 + project_score * 0.60`

Почему так:
- если дать слишком большой вес проблеме, зрелые решения будут теряться;
- если дать слишком большой вес проекту, наверх уйдут технологически красивые, но второстепенные темы.

### Этап 6: Приоритизировать внутри горизонта

Чтобы длинные технологии не проиграли коротким, портфель разделяется на горизонты:

| Горизонт | Смысл |
|---|---|
| `H1` | Эффект в 1-2 года |
| `H2` | Эффект в 2-4 года |
| `H3` | Эффект в 4+ лет |

Сравнение идет **внутри горизонта**, а не только одной общей линейкой.

### Этап 7: Разложить по корзинам управления

| Категория | Диапазон | Управленческий вывод |
|---|---|---|
| `P1` | `4.2-5.0` | Финансировать и продвигать в первую очередь |
| `P2` | `3.5-4.19` | Запускать после `P1` или при наличии ресурса |
| `P3` | `2.8-3.49` | Дорабатывать и готовить к следующему циклу |
| `P4` | `< 2.8` | Не запускать сейчас, пересмотреть позже |

## Инструменты

| Инструмент | Для чего |
|---|---|
| **Паспорт проекта для приоритизации** | Единый нормализованный вход |
| **Таблица оценки проблемного поля** | Problem-first слой оценки |
| **Две модели проектного score** | Разведение direct и enabling |
| **Портфельная карта `H1/H2/H3 x P1/P4`** | Удобная подача для управленческого решения |

## Пример

**Ситуация:** Есть два проекта.

1. Новый химический МУН для карбонатов.
2. Диагностический проект, который резко улучшает выбор кандидатов для ОПР по нескольким активам.

Если сравнивать их только по "ожидаемой нефти", второй проиграет. Если сравнивать problem-first и по своей модели, обеспечивающий проект может подняться выше, потому что снижает неопределенность и влияет на качество большого числа решений.

## Когда метод сложен

- Когда проблемные поля не описаны, а проекты сравнивают "в куче".
- Когда direct и enabling проекты пытаются оценивать по одной шкале.
- Когда нет разбиения по горизонтам, и H3-проекты системно проигрывают H1.

## Связанные принципы

- **OA.OP.001** - KPI и критерии приоритета меняют поведение системы.
- **OA.OP.002** - Потеря контекста проблемы при передаче искажает портфель.
- **OA.IP.001** - Технология сама по себе не равна эффекту.

## Связанные failure modes

- **FM.ORG.001** - Неправильные KPI и критерии приоритета.
- **FM.ORG.002** - Потеря критической информации о проблеме и объекте.
- **FM.IP.001** - Копирование технологии без проверки соответствия проблеме и объекту.
- **FM.IP.003** - Продвижение в масштаб без достаточной зрелости и готовности.
