# Work-Ledger: Шаблон учёта Work переходов конвейера нефти

**Pack:** upstream-petroleum
**FPF Reference:** B.1.6 — Δ_work additivity; B.5.1 — Artifact lifecycle
**Назначение:** Инструмент мониторинга удельных затрат ресурсов (Work) на каждом переходе конвейера нефти. Используется инженером-технологом и владельцем стыка.

> **Связанные файлы:**
> - Register Bridge (формулы мостов): [01D-register-bridge.md](../01-domain-contract/01D-register-bridge.md)
> - Объекты внимания: OA.EW.001, OA.EW.002

---

## Как использовать этот шаблон

1. **Заполните метаданные** (объект, период, ответственный)
2. **Блок А (физика)** заполняется из данных SCADA / приборов учёта — без интерпретации
3. **Блок Б (мост)** заполняется из действующих тарифов / нормативов — указать версию и источник
4. **Блок В (экономика)** рассчитывается по формулам из Register Bridge — не вводится вручную
5. **Диагноз и рекомендация** — всегда с указанием источника (Блок А) и эффекта (Блок В)

## Шаблон: Work-Ledger по одному переходу

### Boundary Ledger Header (CC-B1.6.1)

*Заполняется перед началом измерения. Обязателен для любого результата Δ_work.*

| Поле CC-B1.6.1 | Описание | Значение |
|----------------|----------|---------|
| **Boundary** (граница) | Что является входом и выходом перехода | Вход: ___ / Выход: ___ |
| **Time Window** (временное окно) | Период, за который измеряется Work | дд.мм.гггг — дд.мм.гггг |
| **Basis** (ресурсный базис) | Ссылка на объявленный базис | [01D-register-bridge.md, разд. 2](../01-domain-contract/01D-register-bridge.md) |
| **Method** (метод измерения) | Инструменты и способ получения данных | Счётчик АСКУЭ / ручной замер / расчёт по формуле |
| **Transformer** (кто выполнял) | Субъект, отвечающий за достоверность | ФИО / роль / подразделение |

> *Без заполненного Boundary Ledger Header результат Δ_work не принимается как артефакт-доказательство перехода (INV.001 из bounded context).*

## Эволюционный цикл: работа по снижению OpEx

По FPF B.4 — улучшение η (КПД переходов) ведётся через цикл:

```
Run
 └→ Observe: измерить SW и η по каждому переходу (Блок А)
      └→ Identify: найти переход с наибольшим отклонением от нормы
           └→ Refine: изменить MethodDescription
           │    (другой насос, режим работы, маршрут, реагент, отсечение воды)
           └→ Deploy: ГТМ / замена / регулировка
                └→ Run (новый цикл — проверить, изменилось ли SW и η)
```

**Требование к квалификации:** каждое изменение MethodDescription, претендующее на улучшение η, должно пройти проверку по [UP.M.007-008](../02-domain-entities/) — соответствует ли инновация физическим условиям данного перехода?
