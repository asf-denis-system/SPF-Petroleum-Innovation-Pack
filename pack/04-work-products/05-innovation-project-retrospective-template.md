# WP.RetroProject: Шаблон ретроспективы инновационного проекта (НИР/НИОКР/ОПР)

**Назначение:** Канонический шаблон разбора завершенного инновационного проекта так, чтобы отделять неуспех технологии от неуспеха постановки задачи, испытаний, внедрения и измерения эффекта.

**Версия:** 1.0  
**Статус:** Рабочий шаблон  
**Связано с:** [Шаблоном инновационного проекта](02-innovation-project-template.md), [Моделью трех конвейеров](03-three-pipeline-model.md)

---

## Принципы применения

1. Ретроспектива проводится **поверх фактов и артефактов**, а не как свободный разговор.
2. Результат фиксируется не в бинарной форме `успех / провал`, а в более точном статусе.
3. `5 почему` применяется **к каждому локализованному разрыву**, а не ко всему проекту сразу.
4. Формулировка **"результат не доказан"** не равна формулировке **"технология не работает"**.

## Блок 2: Итоговый статус проекта

Выбрать ровно один итоговый статус:

| Статус | Когда применять |
|---|---|
| **Успех подтвержден** | Эффект подтвержден и есть основание для внедрения или тиража |
| **Частичный успех** | Эффект есть, но только в части условий, объектов или режимов |
| **Гипотеза не подтвердилась, но знание получено** | Гипотеза не дала ожидаемого эффекта, однако сократила неопределенность |
| **Результат не доказан** | Эффект нельзя честно доказать или опровергнуть из-за дизайна испытаний, данных или расчета |
| **Организационный срыв** | Проект сломался по управлению, handover, ресурсам или приоритетам |

Дополнительно заполнить:

| Поле | Заполнение |
|---|---|
| Какой эффект ожидался | |
| Какой эффект фактически получен | |
| Что подтверждено фактами | |
| Что осталось недоказанным | |

## Блок 3: Где порвалась цепочка

Зафиксировать ключевые разрывы в цепочке `проблема -> технология -> испытание -> внедрение -> эффект`.

| Этап цепочки | Был ли разрыв | Тип разрыва | Краткое описание | Артефакт / факт-доказательство |
|---|---|---|---|---|
| Приоритизация проблемы | Да / Нет | | | |
| Выбор технологии / подхода | Да / Нет | | | |
| Инициация проекта | Да / Нет | | | |
| Разработка технологии | Да / Нет | | | |
| Подготовка испытаний | Да / Нет | | | |
| Тестирование / ОПИ / ОПР | Да / Нет | | | |
| Анализ результатов | Да / Нет | | | |
| Принятие решения | Да / Нет | | | |
| Упаковка / внедрение / тираж | Да / Нет | | | |

Типы разрывов удобно использовать такие:
- эффект ниже ожиданий;
- эффект нестабилен;
- результат не доказан;
- неверный объект / условия;
- срыв сроков / бюджета;
- эксплуатация не приняла;
- экономика / масштаб не сошлись;
- смена приоритетов / внешние факторы.

## Блок 4: Классификация причин

Для каждого ключевого разрыва отметить уровень причины и код.

| Код | Уровень причины | Категория | Типовой смысл |
|---|---|---|---|
| A1 | Проблема / целеполагание | Проблема выбрана неверно | Проект стартовал не под реальную приоритетную проблему |
| A2 | Проблема / целеполагание | Критерии успеха не заданы | Нет KPI, baseline или порогов |
| B1 | Гипотеза / концепция | Гипотеза слабая | Идея слабо обоснована физически или методически |
| B2 | Гипотеза / концепция | Ограничения применимости не учтены | Не учтены геология, флюиды, инфраструктура, качество данных |
| C1 | Объект / применимость | Объект не подходит | Объект выбран по доступности, а не по применимости |
| D1 | Разработка / исполнитель | Решение сырое | Прототип или метод не доведен до устойчивого состояния |
| D2 | Разработка / исполнитель | Не хватает компетенций | Ключевая экспертиза или подрядчик были недостаточны |
| E1 | Испытания / данные / расчет | Нет доказательности | Нет контроля, базы сравнения, достаточного окна наблюдения |
| E2 | Испытания / данные / расчет | Данные плохого качества | Недостаток, шум, несопоставимость данных |
| E3 | Испытания / данные / расчет | Атрибуция эффекта некорректна | Эффект посчитан методически неверно |
| F1 | Организация / ресурсы | Недостаточно ресурсов / внимания | Команда, бюджет, время или ритм управления не сработали |
| F2 | Организация / ресурсы | Потерян handover | Критическая информация не дошла до следующей роли |
| G1 | Внедрение / принятие | Эксплуатация не приняла | Не было реального владельца внедрения |
| G2 | Экономика / масштаб | Масштаб не сходится | На масштабе решение не окупается или не удерживается |

## Минимальная spreadsheet-композиция

Если шаблон реализуется в таблице, рекомендуемый состав листов такой:

| Лист | Назначение |
|---|---|
| **Справочники** | Выпадающие значения: тип проекта, класс технологии, статус, этап разрыва, решение |
| **Классификатор_причин** | Коды и описание причин |
| **Реестр_проектов** | Сводная таблица по нескольким проектам |
| **Ретро_форма** | Подробная форма на один проект |
| **Дашборд** | Сводка по статусам, типам проектов и повторяющимся причинам |
